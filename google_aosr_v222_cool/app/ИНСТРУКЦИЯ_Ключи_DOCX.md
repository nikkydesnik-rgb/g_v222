# Руководство по формату ключей в DOCX-шаблонах

## Проблема

Ключи вида `{{Объект строительства}}` с **пробелами** внутри могут не работать, потому что Microsoft Word разбивает текст на несколько XML-элементов (`<w:t>`). Например:

```xml
<!-- Ключ разбит на 3 части -->
<w:t>{{</w:t>
<w:t>Объект строительства</w:t>
<w:t>}}</w:t>
```

При этом парсер не может собрать ключ целиком, и замена не происходит.

---

## Решение: Форматы ключей (100% надёжные)

### 1. snake_case (РЕКОМЕНДУЕТСЯ)

Заменяйте пробелы на **подчёркивания**:

```
{{Объект_строительства}}
{{Организация_застройщик}}
{{Дата_начала_работ}}
{{ФИО_представителя_застройщика}}
```

**Почему это работает:**
- Подчёркивание — это "буквенный" символ
- Word НЕ разбивает текст на отдельные runs по подчёркиванию
- Ключ остаётся цельным в XML

---

### 2. CamelCase

Без пробелов и разделителей, каждое слово с заглавной:

```
{{ОбъектСтроительства}}
{{ОрганизацияЗастройщик}}
{{ДатаНачалаРабот}}
```

**Плюсы:** Компактно, Word не разбивает
**Минусы:** Менее читаемо

---

### 3. Латинская транслитерация

```
{{ObjektStroitelstva}}
{{OrgZastroishik}}
{{DataNachala}}
```

**Плюсы:** Работает в любых системах, нет проблем с кодировкой
**Минусы:** Нужно помнить сокращения

---

### 4. kebab-case (с дефисами)

```
{{Объект-строительства}}
{{Организация-застройщик}}
```

**Плюсы:** Читаемо
**Минусы:** Дефис иногда может вызвать разделение в Word (реже, чем пробел)

---

### 5. Слитное написание (осторожно)

```
{{Объектстроительства}}
```

**Плюсы:** 100% не разобьётся
**Минусы:** Трудно читать

---

## Форматы, которых следует ИЗБЕГАТЬ

### ❌ Пробелы внутри ключа

```
{{Объект строительства}}        ← ПЛОХО
{{Организация - застройщик}}    ← ПЛОХО
{{Организация  –  строитель}}   ← ПЛОХО (тире тоже проблема!)
```

### ❌ Спецсимволы

```
{{Объект/строительство}}        ← ПЛОХО
{{Организация \\ застройщик}}   ← ПЛОХО
{{Ключ "в кавычках"}}           ← ПЛОХО
```

---

## Ключи в таблицах

**Все форматы snake_case работают в таблицах Word.**

Пример таблицы:

| Роль | Организация |
|------|-------------|
| Застройщик | {{Организация_застройщик}} |
| Строитель | {{Организация_строитель}} |
| Проектировщик | {{Организация_проектировщик}} |

**Важно:** Не разбивайте ключ на несколько ячеек. Ключ должен быть целиком в одной ячейке.

## Что изменилось в новой версии

### 1. Smart-парсинг ключей (docxParser.ts)

- Склеивает все `<w:t>` элементы в документе
- Ищет ключи в объединённом тексте
- Поддерживает оба формата: `{{ключ}}` и `<ключ>`

### 2. Smart-замена при заполнении

- Объединяет разбитые на несколько `<w:t>` ключи
- Заменяет всю последовательность одним значением
- Сохраняет форматирование документа

### 3. Предпросмотр как HTML (docxPreview.ts)

- DOCX конвертируется в HTML перед показом
- Браузер корректно отображает HTML
- Работает в iframe без проблем

### 4. Предупреждения в интерфейсе

- Ключи с пробелами помечаются ⚠️
- Показаны рекомендуемые форматы
- Валидация при загрузке шаблона

## Как перейти на snake_case

### Шаг 1: Откройте шаблон Word

### Шаг 2: Найдите все ключи (Ctrl+H)

Найдите: `{{* *}}`

### Шаг 3: Замените пробелы на подчёркивания

Пример замен:
```
{{Объект строительства}}     →  {{Объект_строительства}}
{{Организация - застройщик}} →  {{Организация_застройщик}}
{{ФИО застройщика}}          →  {{ФИО_застройщика}}
```

### Шаг 4: Перезагрузите шаблон в приложении

## Проверка шаблона

После внесения изменений:

1. **Загрузите шаблон** в приложение
2. **Проверьте количество ключей** — должно соответствовать ожидаемому
3. **Заполните данные** — постоянные данные + поля акта
4. **Нажмите "Предпросмотр"** — должен показаться заполненный документ
5. **Нажмите "Скачать"** — DOCX должен содержать заполненные значения

---

## Если проблема остаётся

1. Откройте Developer Tools (F12) → Console
2. Выполните: `window.IDDebug.enable()`
3. Повторите действие, вызвавшее ошибку
4. Выполните: `window.IDDebug.downloadLogs()`
5. Пришлите логи для анализа
